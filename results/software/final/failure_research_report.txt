TASK 1 — WHY IS DDPM ACTOR LOSS DIVERGING?
================================================================================

ROOT CAUSE IDENTIFIED:

Our implementation computes:
  log_pi = -tanh(MSE(noise, predicted_noise) / 10)
  actor_loss = -(log_pi * advantage)

This is WRONG. The correct approach from DDPO (Black et al. 2023, arXiv:2305.13301):

DDPO treats denoising as a multi-step MDP where:
  - State: s_t = (condition, t, x_t)
  - Action: a_t = x_{t-1} (the denoised sample)
  - Policy: pi(a_t|s_t) = p_theta(x_{t-1} | x_t, c) — an isotropic Gaussian
  - Reward: r(x_0, c) only at final step t=0

The exact log-likelihood for each denoising step is:
  log p_theta(x_{t-1} | x_t, c) = -1/(2*sigma_t^2) * ||x_{t-1} - mu_theta(x_t, t, c)||^2

This is NOT the noise prediction MSE. It's the RECONSTRUCTION error at each step,
weighted by the inverse variance of the posterior.

The DDPO gradient is:
  nabla_theta J = E[sum_{t=0}^{T} nabla_theta log p_theta(x_{t-1}|x_t, c) * r(x_0, c)]

Key insight: the reward r is the SAME for all denoising steps in a trajectory.
Only the log-probabilities change per step.

WHAT WE'RE DOING WRONG:
1. We compute log_pi from noise prediction MSE, not reconstruction log-likelihood
2. We apply reward weighting to a SINGLE denoising step, not all T steps
3. We use tanh normalization which destroys the gradient signal
4. Advantage should be computed from the full trajectory return, not single-step

CORRECT IMPLEMENTATION (from DDPO paper):
```python
# For each denoising step t in trajectory:
# 1. Compute mu_theta = predicted clean sample from x_t
# 2. Compute log_prob = -||x_{t-1} - mu_theta||^2 / (2 * beta_t)
# 3. Sum log_probs across all steps = total log_prob for trajectory
# 4. actor_loss = -total_log_prob * reward  (REINFORCE)
# Or with importance sampling ratio for multiple optimization steps (DDPO-IS)

# DDPO-SF (score function / REINFORCE):
loss = sum(log_prob_t for t in range(T)) * reward

# DDPO-IS (importance sampling, like PPO):
ratio = prod(new_prob_t / old_prob_t for t in range(T))
loss = ratio * reward
# With PPO clipping on the ratio
```

REFERENCE IMPLEMENTATION:
https://github.com/jannerm/ddpo
Tested on Google Cloud TPUs. Not tested on GPU.

RELATED PAPERS:
- DPPO (Diffusion Policy Policy Optimization): arxiv.org/abs/2409.00588
  - Best practices for fine-tuning diffusion policies in continuous control
  - Uses multiple denoising steps per gradient update
  - Key: clip the probability ratio across ALL denoising steps
- TDPO-R: Temporal Diffusion Policy Optimization with critic Reset
  - github.com/ZiyiZhang27/tdpo
  - Addresses reward overoptimization in diffusion RL
  - Key: dormant neurons regularize against overoptimization

RECOMMENDED FIX:
Replace our custom Eq. 33 implementation with DDPO-IS (importance sampling):
1. Collect full denoising trajectories (all N steps)
2. Compute log_prob for each step using Gaussian likelihood
3. Compute importance ratio = new_log_prob / old_log_prob
4. Apply PPO clipping to the ratio
5. Multiply by advantage (normalized)

This is fundamentally different from what we're doing. Our current approach
computes noise MSE at a SINGLE midpoint step and multiplies by advantage.
DDPO uses the FULL trajectory likelihood with importance sampling.

TASK 3 — COMPRESSION RATIO: 0.93 vs 0.73
================================================================================

DeepSC compression ratio definition (Xie et al. 2021, IEEE TSP):
  "The number of transmitted symbols per word"
  rho = k / L_s
  where k = number of complex channel symbols transmitted
        L_s = number of words in the sentence

In our DeepSC model:
  - Semantic encoder output: [batch, seq_len, d_model] = [1, L, 128]
  - Channel encoder output: [batch, seq_len, 16] = [1, L, 16]
  - Each word maps to 16 channel symbols
  - rho = 16 / 1 = 16 symbols per word

Wait — that's 16x EXPANSION, not compression!

Looking at the DeepSC paper more carefully:
  "Fig. 8 illustrates the impact of the number of symbols per word
   on the 1-gram BLEU score when SNR is 12 dB."

The paper tests different symbols-per-word ratios (4, 8, 16, 32).
With 16 symbols/word at SNR=12dB, BLEU ≈ 0.8.

The compression ratio in the Sun et al. 2026 paper (our reference):
  "CSCA achieves compression ratios of 73.3%, 32.2%, and 21.3%
   on text, audio, and image modalities respectively"

This is NOT DeepSC's compression ratio. This is MSS (Minimum Synonymous
Subsequence) compression: the ratio of simplified text length to original.

So the paper's 73% text compression comes from MSS with eta=0.85,
NOT from DeepSC's channel encoding. Our MSS achieves 93% because:
- Our eta=0.85 is too permissive (keeps more words)
- The paper's MSS may use a lower eta threshold
- Or the paper's LAM generates shorter descriptions than our SST sentences

FIX: Lower eta in source_simplifier.py from 0.85 to 0.6-0.7.
This would reduce compression from 0.93 to closer to 0.73.

Also: the paper's compression is measured on LAM-generated descriptions
of multimodal data, NOT on raw SST sentences. LAM descriptions are
inherently longer and more redundant, so MSS removes more words.

Our test uses SST sentences which are already concise. That's why
MSS keeps 93% — there's less redundancy to remove.
